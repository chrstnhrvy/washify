1. Project Overview

Washify is a lightweight, multi-tenant SaaS-style laundry manager. Any shop owner signs in with Google, gets their own private workspace, logs orders, notifies customers when their laundry is done (via SMS or Facebook Messenger), tracks sales, and answers customer questions with an AI chatbot. It runs entirely on free tiers.

The product is real, but its purpose is to demonstrate three competencies end-to-end:

  • Workflow automation — event-driven n8n workflows behind webhooks.
  • Applied AI — a Retrieval-Augmented Generation (RAG) chatbot and an LLM message-drafting feature.
  • Real system design — a multi-tenant database with security enforced in the database, not the UI.

The problem it models

A small laundromat has three recurring pains: customers asking "is it ready yet?", staff manually texting people, and the same FAQs (hours, price, turnaround) being asked all day. Washify automates all three and keeps every shop's data isolated, so it behaves like a true SaaS product.

2. System Architecture

   Public visitor ─────▶  Landing page (Vercel, React SPA) + FAQ chat widget
                                         │ POST /faq-chat
   Shop owner ──Google OAuth──▶  /app (auth-gated workspace)
        │                        orders · sales · customers · settings
        │ read / write (scoped by RLS)            │ POST /laundry-done, /messenger-send, /draft-message
        ▼                                          ▼
   ┌──────────────────────────┐         ┌──────────────────────────────┐
   │ Supabase Postgres         │         │ n8n (Docker, exposed via ngrok)│
   │  shops / orders /          │◀────────│  webhooks + Groq + Gemini      │
   │  customers + RLS + pgvector│         │  + SMS API PH + Facebook Graph │
   └──────────────────────────┘         └──────────────────────────────┘

Why this shape: the React app talks directly to Supabase using the signed-in user's token; Row Level Security guarantees isolation, so there is no custom backend to maintain. Anything that is "automation" (sending an SMS, answering with AI, drafting a message) is pushed behind an n8n webhook. That separation is the core design decision: the app never holds an API key, and a channel can be swapped by editing one n8n node without touching the app.

4. n8n Concepts — Webhook vs HTTP Request, GET vs POST

This is the section most worth understanding cold, because two pairs of ideas are easy to confuse.

Webhook node vs HTTP Request node

Webhook nodeHTTP Request node
DirectionReceives — n8n is the server, waiting to be calledSends — n8n is the client, calling someone else's API
AnalogyA door someone knocks onA phone call n8n makes
Position in a workflowAlways the start (a trigger)A step in the middle
In WashifyApp / Meta call into n8n (/laundry-done, /faq-chat, etc.)n8n calls out to SMS API PH and Facebook

So a single workflow often has both: a Webhook receives the request, then an HTTP Request calls an external service to do the work.

GET vs POST (the HTTP methods)

GETPOST
PurposeRetrieve / read dataSend / create / trigger an action
Where data ridesIn the URL query string (?key=value)In the request body (usually JSON)
Has a body?NoYes
Repeat-safe?Yes (idempotent, read-only)No (each call may create/change something)

How that combines in Washify

  • Webhook · POST The normal case — the app sends a JSON body to n8n (e.g. { phone, message }). Used by every app-triggered workflow.
  • Webhook · GET Used once, for Facebook's webhook verification. Meta sends a GET with a hub.challenge query value; n8n must echo it back as plain text to prove it owns the URL. No body involved — that's why it's a GET.
  • HTTP Request · POST n8n sending data out: posting the SMS to SMS API PH, or posting the message to the Facebook Graph Send API.
  • HTTP Request · GET Would fetch data from an external API (not needed in the current workflows, since every external call creates something — a sent message).
Test URL vs Production URL. While editing, n8n gives a /webhook-test/… URL that listens for one call after you click Execute. Once you Activate the workflow, the permanent /webhook/… URL works continuously. The app always calls the production URL.

Other n8n building blocks used

  • Edit Fields (Set) — builds or reshapes data (e.g. normalize a phone number, pull a value out of a nested payload).
  • Respond to Webhook — sends the HTTP response back to whoever called the webhook (so the app gets { status: "sent" }).
  • Credentials — n8n's encrypted store for API keys. Nodes reference a credential by name, so exported workflow JSON never contains the real key.
  • AI sub-nodes — Chat Model, Embeddings, Vector Store and Data Loader attach underneath a chain node to give it a brain, a memory, and a retriever.

5. The n8n Workflows, Node by Node

Washify uses six workflows. Each is introduced with what it does, then broken down node by node.

a. Laundry-Done SMS Notifier (washify-sms)

What it does: when staff tap "Text" on a ready order, the app posts the phone + message and this workflow sends a real SMS. Flow: Webhook → Edit Fields → HTTP Request → Respond.

d. Messenger Inbound — capture & link PSID (n8n-fb-message)

What it does: handles Facebook's verification and incoming messages, and links a customer's Messenger ID (PSID) to an order. It is two independent chains on one path.

NodeTypeWhat it does
VerifyWebhook (GET /messenger)Receives Meta's verification GET.
Respond to WebhookRespond (text)Echoes back hub.challenge to prove URL ownership.
IncomingWebhook (POST /messenger)Receives every message the Page gets.
Extract PSIDSetPulls out the sender's PSID and normalizes the message text into an order code.

Why match on the typed code instead of a hidden referral? Facebook's m.me?ref= referral events are unreliable and need extra subscriptions. The message text always arrives, so matching the order code the customer sends is far more dependable.

What it does: sends a Messenger message, mirroring the SMS workflow. Flow: Webhook → HTTP Request → Respond.

6. How the AI / RAG Pipeline Works

Retrieval-Augmented Generation means: fetch the relevant facts first, then let the model answer from them — so it stays grounded and doesn't invent policies.

  1. Ingest (once / on change): split the FAQ into chunks → turn each into a 768-dimension vector with Gemini → store in Supabase pgvector.
  2. Query (every question): embed the question → cosine-similarity search for the closest chunks → hand them to Groq with a strict system prompt → return the answer.

Why two AI providers? Groq is fast and free for generation; Gemini's text-embedding-004 is free for embeddings (Groq doesn't offer embeddings). Using the right tool for each half is a deliberate, cost-aware choice.

The AI message drafting feature uses the same Groq model but no retrieval — it's a single prompt that turns structured order data into a warm, length-limited message.

7. Multi-Tenant Security & Secrets

Row Level Security (RLS)

Every owner is one shop, keyed so shops.id = auth.uid(). Each row in orders and customers carries a shop_id, and RLS policies allow access only where shop_id = auth.uid(). Isolation is enforced in the database, so even a forged client request cannot read another shop's data. A Postgres trigger auto-creates a shop row on first sign-in.

Secret handling

  • API keys live only in n8n's encrypted credential store (or, for the frontend, the Supabase anon key, which is safe to expose because RLS protects it).
  • Workflow JSONs committed to the repo use placeholders (YOUR_API_KEY), never real keys.
  • Lesson learned: a key committed once stays in git history — redacting later isn't enough, you must rotate it.

8. SMS vs Messenger — Channel Constraints

SMS (SMS API PH)Facebook Messenger
Recipient identified byPhone number (already collected)PSID (only obtained after the customer messages the Page)
Can you message cold?YesNo — customer must opt in first
Timing limitNone (1 SMS / 10 s rate limit)24-hour window after the customer's last message
Message length160 chars (1 segment)2000 chars
CostFree gatewayFree, but policy-restricted

Key insight: SMS needs no opt-in, so it's the primary channel for "your laundry is ready." Messenger is an opt-in extra. Matching the channel to its constraint is the engineering point.

9. The Application (Frontend)

React + TypeScript SPA, organized feature-first (features/landing, orders, dashboard, customers, settings, auth, faq-chat). The authed workspace and heavy libraries are code-split, so public visitors download almost none of the app code.

Post-login features

  • Orders — add (with customer auto-fill), edit, delete, status pipeline (Received → Washing → Ready → Picked up), paid toggle, search + filter.
  • Notify — SMS or Messenger, with an "AI draft" button and a Messenger connect QR.
  • Sales dashboard — revenue, loads/kg, orders, outstanding balance, average order value, repeat-customer rate, week-over-week change, charts, and Excel export.
  • Customers — directory auto-built from orders, with visit counts.
  • Printable claim stub — order code + Messenger QR.
  • Settings — shop name, per-load / per-kilo pricing, price, Facebook Page ID.
  • Realtime — orders update live via Supabase Realtime (e.g. a PSID captured by n8n appears without a refresh).

11. Challenges & How They Were Solved

  • Empty webhook response crashed the app. An SMS call returned 200 with an empty body; JSON.parse("") threw. Fix: the n8n client now tolerates empty/non-JSON responses.
  • Leaked API key. A key was committed in a workflow JSON and caught by a secret scanner. Fix: redact to a placeholder, keep real keys only in n8n, and rotate the exposed key.
  • Deprecated Messenger tags. Sending with POST_PURCHASE_UPDATE failed. Fix: send messaging_type: RESPONSE inside the 24-hour window.
  • Unreliable referral links. m.me?ref= often returned a null ref. Fix: switched to matching on the order code in the message text, which always arrives.
  • Version-sensitive AI nodes. n8n's LangChain nodes shift between releases, so RAG workflows are best built node-by-node and verified, not blindly imported.

12. Glossary

TermMeaning

RLS (Row Level Security)Database rules that restrict which rows a user can read/write.

13. Interview Talking Points & Likely Questions

What this project demonstrates

  • Automation: designed and debugged real n8n workflows with webhooks, data transforms, external API calls, and error paths.
  • Applied AI: a working RAG pipeline (vector DB + embeddings + LLM + prompt design) and an LLM drafting feature.
  • Integration thinking: app and automations are decoupled behind webhooks, so either side can change independently.
  • System design: multi-tenant data with database-enforced isolation.
  • Judgement: chose the right channel per constraint, and dropped a flaky approach (referral) for a reliable one (code-in-message).

Questions an interviewer might ask

QuestionHow to answer
Why n8n instead of writing a backend?It keeps automation visual and swappable, keeps API keys off the frontend, and is the exact skill the role targets. The app stays a thin client.
How do you keep one shop from seeing another's data?Row Level Security in Postgres — every query is filtered by shop_id = auth.uid() in the database, not the UI.
How does the chatbot avoid making things up?RAG: it only answers from retrieved FAQ chunks, with a system prompt that says "if it's not in the context, say you're not sure."
Difference between the Webhook and HTTP Request nodes?Webhook receives (n8n is the server / trigger); HTTP Request sends (n8n is the client, mid-workflow).
Why SMS and Messenger both?SMS works cold (no opt-in); Messenger is free and richer but needs opt-in + a 24-hour window. Channel matched to constraint.
What was the hardest bug?The Messenger PSID linking — referral events were unreliable, so I switched to matching the order code in the message text.
How would you scale the chatbot to many shops?Filter the vector search by a shop_id in document metadata so each shop only retrieves its own FAQ.
